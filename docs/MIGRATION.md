# Migration Plan

## 移行原則

- 旧リポジトリは移行完了まで変更・削除・アーカイブしない
- 計算エンジンの履歴を失う単純コピーは行わない
- 現行固定コミット `0f91ba8381df1b4960557b92b39339385cc9009f` を基準に結果一致を確認する
- 各段階でColab版を実行可能な状態に保つ

## Phase 1: 基盤（完了）

- モノレポ構成
- 共通入力スキーマの型
- JSON実行用Gradioアプリ
- 最小CI
- 外部Git依存として現行計算エンジンを固定

## Phase 2: 入力スキーマ移行（完了）

完了済み:

- 現行222項目を機械抽出し、バージョン付きJSON台帳へ保存
- 基本設定45・暖房84・冷房91・換気2の内訳を契約テストで固定
- 18セクション、初期値、選択肢、重複変数の出現番号を固定
- ⑦-1〜⑦-4、⑧-1〜⑧-4の方式条件を台帳へ格納
- UI変数から最終 `input_data` への264代入を、条件分岐と元行番号を含めて対応付け
- 222項目の対応漏れ・未知の入力IDがないことを契約テストで固定
- 現行フォームから同じ `input_data` を生成できる互換ビルダーを追加

- 互換ビルダーの `exec` を廃止し、現行フォーム専用の制限付きAST評価へ置換
- 264代入の値式と条件式を型付きの宣言的変換へ置換
- 台帳を222項目の正規 `InputSchema` へ変換
- ラベル・セクション・グループ・型・選択肢・デフォルト値を正規スキーマへ保持
- デフォルト、全暖冷房方式、全boolean・select代替値で現行 `input_data` との完全一致を契約テストで固定
- 計算エンジンに入力DTOがないため、最終 `dict[str, Any]` の完全一致を境界契約として採用

## Phase 3: Gradio移行（完了）

完了済み:

- Base64埋め込みコードを廃止
- 共通スキーマからフォームを生成
- 計算ログ、グラフ、ダウンロードをサービスへ分離
- Colabノートブックをインストールと起動だけに縮小

## Cloud Migration Track（完了）

- Gradioアプリをコンテナ化し、Cloud Runの `0.0.0.0:8080` 契約に対応
- 計算出力を `VERIFICATION_OUTPUT_DIR` 配下へ隔離し、Cloud Runでは一時領域を使用
- 状態を共有する計算エンジンに合わせ、Cloud RunとGradio双方の同時実行数を1に制限
- GitHub ActionsのWorkload Identity Federationで、長期サービスアカウント鍵を置かずにデプロイ
- Cloud Runサービス定義にCPU、メモリ、タイムアウト、最大インスタンス数を明示

## Phase 4: 計算エンジン履歴移行（完了）

`git subtree` を使用して、`pyhees-jjj` の固定コミット `0f91ba8381df1b4960557b92b39339385cc9009f` までの履歴を `packages/pyhees-jjj` へ統合しました。

アプリの計算エンジン依存とコンテナビルドを、モノレポ内のローカルパス依存へ変更しました。

## Phase 5: 回帰確認と切替

- 代表入力JSONでCSV出力を旧版と比較
- 浮動小数点の許容誤差を明示
- Colab、ローカル、コンテナの動作確認
- 切替完了後に旧リポジトリを参照専用化
