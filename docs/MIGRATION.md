# Migration Plan

## 移行原則

- 旧リポジトリは移行完了まで変更・削除・アーカイブしない
- 計算エンジンの履歴を失う単純コピーは行わない
- 現行固定コミット `0f91ba8381df1b4960557b92b39339385cc9009f` を基準に結果一致を確認する
- 各段階でColab版を実行可能な状態に保つ

## Phase 1: 基盤（完了）

- モノレポ構成
- 共通入力スキーマの型
- JSON実行用Gradioアプリ
- 最小CI
- 外部Git依存として現行計算エンジンを固定

## Phase 2: 入力スキーマ移行（進行中）

完了済み:

- 現行222項目を機械抽出し、バージョン付きJSON台帳へ保存
- 基本設定45・暖房84・冷房91・換気2の内訳を契約テストで固定
- 18セクション、初期値、選択肢、重複変数の出現番号を固定
- ⑦-1〜⑦-4、⑧-1〜⑧-4の方式条件を台帳へ格納
- UI変数から最終 `input_data` への264代入を、条件分岐と元行番号を含めて対応付け
- 222項目の対応漏れ・未知の入力IDがないことを契約テストで固定
- 現行フォームから同じ `input_data` を生成できる互換ビルダーを追加

次に行うこと:

- 文字列の代入式・条件式を型付きの宣言的変換へ段階的に置換
- 台帳を正規 `InputSchema` へ変換
- デフォルトおよび代表的な暖冷房方式で、現行ノートブックの `input_data` と完全一致する契約テストを追加
- ラベル・説明・デフォルト値の正本を計算エンジン側DTOと照合

## Phase 3: Gradio移行

- Base64埋め込みコードを廃止
- 共通スキーマからフォームを生成
- 計算ログ、グラフ、ダウンロードをサービスへ分離
- Colabノートブックをインストールと起動だけに縮小

## Phase 4: 計算エンジン履歴移行

`pyhees-jjj` の対象履歴を `packages/pyhees-jjj` へ履歴付きで移行します。推奨手段は `git filter-repo` または `git subtree` です。

履歴移行後に、アプリのGit依存をローカルパス依存へ変更します。

## Phase 5: 回帰確認と切替

- 代表入力JSONでCSV出力を旧版と比較
- 浮動小数点の許容誤差を明示
- Colab、ローカル、コンテナの動作確認
- 切替完了後に旧リポジトリを参照専用化
